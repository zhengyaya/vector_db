## 阶段1 完成后的问答

### Q1: Chroma 数据库的表结构和作用

**问题**：通过 DB Browser for SQLite 查看存储的内容，请讲解数据所有的表的作用和内容

**解答**：

Chroma 使用 SQLite 存储数据，共创建了 **21 个表**。核心表如下：

| 表名 | 作用 | 记录数 | 说明 |
|------|------|--------|------|
| **collections** | 集合定义 | 1 | 存储集合名称、维度（384维向量）、配置信息 |
| **embeddings** | 向量索引 | 45 | 记录每个文档块的 ID 和 seq_id 映射关系 |
| **embedding_metadata** | 元数据 | 90 | 存储文档内容和来源（每个文档2条：source + content） |
| **segments** | 数据分段 | 2 | 定义向量存储段和元数据存储段 |

**关键发现**：
- 45 个文档块，每块约 500 字
- 向量维度 384（Chroma 默认的中文 embedding 模型维度）
- 每个文档有2条元数据：source（来源）+ chroma:document（内容）
- 向量和内容分开存储：vectors 在 segments，内容在 embedding_metadata

### Q2: Chroma 与传统关系型数据库的关系

**问题**：Chroma 是在传统关系型数据库基础上，增加了向量的能力么？

**解答**：

**完全正确！**

```
┌─────────────────────────────────────────────────┐
│                  Chroma                         │
├─────────────────────────────────────────────────┤
│  传统数据库能力          向量能力              │
│  ├─ SQLite 存储         ├─ 文本→向量 embedding │
│  ├─ 表结构（collections）├─ HNSW 向量索引      │
│  ├─ SQL 查询            ├─ 相似度检索         │
│  ├─ 元数据管理          ├─ 向量距离计算        │
│  └─ 事务/持久化         └─ ANN 近似搜索        │
└─────────────────────────────────────────────────┘
```

| 维度 | 传统数据库 | Chroma |
|------|----------|--------|
| 存储 | 结构化数据 | 向量 + 结构化 |
| 查询 | SQL精确匹配 | 向量相似度检索 |
| 索引 | B+Tree | HNSW |
| 用途 | 精确查找 | 语义搜索 |

> **本质**：在传统键值存储上封装了一层向量语义搜索能力。

### Q3: 向量数据的存储位置

**问题**：实际参与语义检索的向量是存在 bin 二进制文件中，通过 DB Browser 只能看到向量索引关系、文档片段的原始内容，是吗？

**解答**：

**完全正确！**

| 数据类型 | 存储位置 | DB Browser 能看到？|
|----------|----------|-------------------|
| **向量数据** | `data_level0.bin` 等 bin 文件 | ❌ 无法查看 |
| **文档内容** | `embedding_metadata` 表 | ✅ 可见 |
| **索引关系** | `embeddings` 表 | ✅ 可见 |
| **元数据** | `collection_metadata` 表 | ✅ 可见 |

文件分布：
```
chroma_data/
├── chroma.sqlite3              ← 元数据（DB Browser 可看）
└── 706d77a0-3fee-4965-af9b-1f421c2121cf/
    ├── data_level0.bin         ← 向量数据（二进制）
    ├── header.bin
    ├── length.bin
    └── link_lists.bin
```

> **结论**：DB Browser 看到的是"元信息"（谁对应谁、内容是什么），真正的"向量数值"存储在 bin 文件中，由 Chroma 的 HNSW 索引管理。

### Q4: 向量数据的计算公式

**问题**：解读 `45个 × 384 × 4字节 ≈ 69KB`

**解答**：

```
45 个文档块 × 384 维 × 4 字节 = 69,120 字节 ≈ 69KB
```

| 部分 | 含义 |
|------|------|
| **45** | 你的文档被切分成了 45 个文本块（chunk）|
| **384** | 每个文本块转换成的向量维度（Chroma 默认中文模型）|
| **4 字节** | float32（32位浮点数）的固定大小 |

**为什么是 384 维？**
- 这是 Chroma 内置的 **default** embedding 模型（基于 sentence-transformers）的输出维度
- 文本 "什么是多 Agent 协同？" → embedding 模型 → [0.12, -0.34, 0.56, 0.01, ..., 0.23] ← 384 个数字

**float32 是什么？**
| 类型 | 精度 | 字节 |
|------|------|------|
| float32 | 32位（7位有效数字）| 4 |
| float64 | 64位（15位有效数字）| 8 |

Chroma 用 float32 足够了 —— 向量只需"近似匹配"，不需要极高精度。

**图示理解**：
```
一个文本块（如第1段）
        ↓
   ┌────────────────────────────────────────┐
   │ embedding 模型                        │
   └────────────────────────────────────────┘
        ↓
   [0.1] [0.2] [0.3] ... [0.384]  ← 384个数
    ↑      ↑      ↑          ↑
    4B     4B     4B         4B  ← 每个占4字节
    ↓      ↓      ↓          ↓
   ┌─────────────────────────────┐
   │    data_level0.bin          │  ← 45个这样的向量
   │  45 × 384 × 4 = 69KB    │
   └─────────────────────────────┘
```

> **关键理解**：向量就是一组数字（数组），维度越高，信息越多，但存储/计算开销也越大。384 维是平衡点（不算太大，又能捕捉语义）。

## 阶段2：RAG 知识问答系统

### RAG 架构概览

```
用户问题
    ↓
1. 向量检索（Chroma）→ 召回 Top K 相关文档
    ↓
2. 构建 Prompt → 拼接上下文
    ↓
3. 调用 LLM（M3 模型）→ 生成答案
    ↓
返回答案 + 参考来源
```
